Annotation Token Approaches

How annotations are written in the editor and rendered in the song view

{c: Chorus} {a: xx and yy to sing this}
[Am]Amazing [F]grace how [C]sweet the [G]sound
[Am]That saved a [F]wretch like [C]me {a: sing softly}
↓ renders as
Chorus — xx and yy to sing this
Amazing grace how sweet the sound
That saved a wretch like me — sing softly

A — {a: text} inline suffix

Annotation token appended to the same line it annotates. Consistent with existing {c:} syntax. Works on section headers and any content line.

Pros

  • One token, three levels
  • Visually clear in editor
  • Easy to parse

Cons

  • Long annotations clutter the line
{c: Chorus}
{note: xx and yy to sing this}
[Am]Amazing [F]grace how [C]sweet the [G]sound
[Am]That saved a [F]wretch like [C]me
{note: sing softly}
↓ renders as
Chorus
xx and yy to sing this
Amazing grace how sweet the sound
That saved a wretch like me
sing softly

B — {note: text} on its own line

Annotation lives on a separate line immediately after the element it annotates. Renders as a small note line between elements.

Pros

  • Doesn't break long lines
  • Can be long notes

Cons

  • Ambiguous which line it annotates
  • Harder to parse correctly
{c: Chorus} // xx and yy to sing this
[Am]Amazing [F]grace how [C]sweet the [G]sound
[Am]That saved a [F]wretch like [C]me // sing softly
↓ renders as (same as A)
Chorus — xx and yy to sing this
Amazing grace how sweet the sound
That saved a wretch like me — sing softly

C — // comment style suffix

Familiar to programmers. Double-slash starts an annotation anywhere on a line. Simpler to type but less consistent with the existing {c:} token style.

Pros

  • Fastest to type
  • Visually distinct

Cons

  • Conflicts if "//" appears in lyrics
  • Inconsistent with existing syntax

Song-level annotation (shown near the title) would be a dedicated field in the song metadata editor for all three options.
