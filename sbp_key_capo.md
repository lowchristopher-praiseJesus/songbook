# How SongBook Pro encodes key, KeyShift, and Capo

**The `.sbp` file stores three independent numeric fields that together produce what the player sees on screen: `key` is the song's original written/sounding key as a 0–11 chromatic integer (MIDI pitch-class: C=0, F=5, A♯/B♭=10), `KeyShift` is a live semitone transposition applied non-destructively at render time, and `Capo` is a guitar fret number that further lowers the *displayed chord shapes* so a physical capo at that fret produces the target sounding key.** This matters for anyone reading or writing `.sbp` files programmatically because the stored chord tags inside `content` are almost always in the original key — the app computes the display on the fly by combining all three fields. SongBook Pro (developer: Jonathan Salmon / Songbook Systems Ltd., domain `songbook-pro.com`, not to be confused with LinkeSoft SongBook or MySongbook) publishes a user manual describing the UI behavior but no file-format specification, so the integer encodings below are inferred from the documented behavior, a third-party reverse-engineering writeup, and the conventions of directly comparable apps (ChordPro, OnSong, OpenSong). The result is a model that cleanly explains every value seen in real `.sbp` files, including the user's observed `key=5`, `key=10`, `KeyShift=0`, `Capo=0`.

## Container structure and where the fields live

A `.sbp` (or `.sbpbackup`) file is a plain ZIP archive containing exactly two entries: **`dataFile.txt`** and **`dataFile.hash`**. The text file begins with the literal version line `1.0\r\n` followed by a single-line JSON object of the form `{"songs":[…]}`. The companion `.hash` file holds the 32-character MD5 hex digest of `dataFile.txt`, used by the app as an integrity check. This structure was confirmed by Frans-Willem Post's 2020 reverse-engineering writeup and by Steve's SongbookPro File Examiner, and both tools are read-only inspectors — no official schema has ever been published. Every song object in the `songs` array carries its own `key`, `KeyShift`, and `Capo` integers alongside the chord-bearing `content` string.

## The `key` field: original key as MIDI pitch class

**The `key` integer is a 0–11 chromatic pitch class using the standard MIDI convention**, where **0=C, 1=C♯/D♭, 2=D, 3=D♯/E♭, 4=E, 5=F, 6=F♯/G♭, 7=G, 8=G♯/A♭, 9=A, 10=A♯/B♭, 11=B**. The observed values map directly: `key=5` is **F**, `key=10` is **B♭** — both extremely common song keys, especially in worship and pop repertoire. No alternative convention (circle-of-fifths index, sharp-count, etc.) produces musically plausible keys across the full 0–11 range, and every comparable music library (music21, Tonal.js, Web MIDI, VexFlow) uses this same encoding.

Semantically, `key` represents the **song's original/written key** — the metadata the song was authored in, not the current display key. The SongBook Pro editor explicitly labels this as "Original Key" in the song-editing screen (visible in the official docs screenshot on the Editing Songs page), and the import flow asks for a key for every song, with a "Detect" option that infers it from the chord text if left blank. The `{key: …}` ChordPro directive is explicitly an **import-only** directive in SongBook Pro's supported-directive list, which means the value is read into this numeric field at import time and written back out at export time, but editing it in the app requires the Song Editor, not the live transpose control.

A single 0–11 integer cannot, on its own, distinguish major from minor. Release notes confirm SongBook Pro only began supporting minor keys in v19.5/20.0 (late 2019 / early 2020), which strongly suggests the major/minor flag is stored as a **separate field** in the JSON (likely a boolean adjacent to `key`, though the exact name is undocumented). The user's research scope excludes minor-key edge cases, so for practical purposes treat `key` as the pitch-class of the tonic.

## The `KeyShift` field: live, non-destructive transposition

**`KeyShift` is a signed integer count of semitones representing the user's live transposition on top of `key`.** `KeyShift=0` is the neutral/untransposed state. When a user opens the Quick Actions menu and changes the "Key" dropdown, the app does **not** rewrite the `[Chord]` tags inside `content` — it stores a shift and applies it at render time. Every user account of transposing in SongBook Pro ("transpose on the fly… then transpose back when done" — Andrew Tuline; "Transpose into any key instantly" — Google Play review) describes this as a reversible, display-layer operation, which is only possible if the original chord text remains intact in `content`.

The displayed sounding key is therefore **`(key + KeyShift) mod 12`**. If a song is stored with `key=2` (D) and the user picks G from the transpose dropdown, the app stores `KeyShift=5` and re-letters every chord five semitones up at display time; the `content` string still contains the original D-key chord tags. This also explains why separate `key` and `KeyShift` fields exist at all — if `key` were meant to mutate with user transpositions, `KeyShift` would be redundant. The same split is used by OnSong ("Key" vs. live Transpose), OpenSong (Original Key vs. Transpose feature), and ChordPro (`{key}` vs. `{transpose}`).

There is **one important exception**: since version 19.0 (January 2019), an editor setting lets users opt in to "edit songs in their transposed key," which **does** rewrite the chord tags persistently in `content` and presumably resets `KeyShift` to 0 in the process. By default, however, transposition is purely a display-time shift. The same non-destructive model handles both the inline `[Chord]` bracket notation and the "chords-above-lyrics" format — the app parses chord tokens in both styles and re-letters them identically on render.

## The `Capo` field: displayed chord shapes, not sounding pitch

**`Capo` is an integer fret number, typically 0–11, with 0 meaning no capo.** The official Quick Actions documentation states the behavior unambiguously: *"Capo/transpose — Use this control to set the capo you want to use in the song. The chords will adjust such that when you play the chords with that capo, they will be in the key set in the box above."* This is the single most important sentence in the manual. It means SongBook Pro shows the **chord shapes the guitarist physically plays** (option **a** from the user's question), not the concert-pitch/sounding chords — if the target key is D and `Capo=2`, the on-screen chords are the C-family shapes (C, F, G, Am, etc.), and the guitarist gets D sounding because of the physical capo.

In display terms, the chord letters shown are shifted **down** from the sounding key by `Capo` semitones: **displayed shape = (key + KeyShift − Capo) mod 12**, while the **sounding key** remains `(key + KeyShift) mod 12`. Both values appear simultaneously in the app's UI — Tuline's screenshots show the top-right corner of a song displaying a format like "D / Capo 2 (C)", meaning "sounds in D, played with C shapes at capo 2." So yes, the app shows both original/sounding and played keys at once, and the Song Editor separately exposes the stored "Original Key" label.

`Capo` and `KeyShift` are therefore **independent** fields that combine additively: `KeyShift` moves the sounding key, and `Capo` then lowers the displayed chord letters by the fret count without affecting the sounding key. A subtlety confirmed by the Windows legacy release notes: when the user switches the instrument away from guitar, the UI relabels the "Capo" control as "Transpose" and allows **negative** values, which strongly implies the underlying integer is already a semitone offset stored with a signed type — on guitar it's just constrained to non-negative fret numbers.

## How the three fields combine at render time

The display pipeline for any chord token `[X]` inside `content` is effectively a three-stage transposition:

| Stage | Operation | Net effect on chord letter |
|---|---|---|
| Stored | Chord `X` in original `key` | Chord named for the written key |
| After `KeyShift` | Shift up by `KeyShift` semitones | Now in the user's chosen sounding key |
| After `Capo` | Shift down by `Capo` semitones | Now in the capo-shape key the guitarist fingers |

So the **displayed chord = transpose(X, KeyShift − Capo)**, the **sounding key = (key + KeyShift) mod 12**, and the **displayed key indicator = (key + KeyShift − Capo) mod 12**. The observed user values (`Capo=0`, `KeyShift=0`) mean no transformation is applied at all — the chord tags in `content` render exactly as written, and `key` labels the song in its stored pitch. This is the common case for songs that haven't been touched since import.

Two practical implications for programmatic consumers of `.sbp` files: first, to learn what a song *sounds* like you read `key + KeyShift` and ignore `Capo`; to reproduce what the on-screen chord chart *looks* like you need all three. Second, because transposition is normally non-destructive, the canonical representation of a song's chord progression lives in `content` with chord tags relative to `key` — a round-trip through SongBook Pro with transpose or capo adjustments should leave `content` byte-identical unless the user explicitly used the "edit in transposed key" feature.

## Conclusion and unresolved details

SongBook Pro's three-field model is cleanly orthogonal: `key` identifies the song, `KeyShift` moves the sounding pitch, `Capo` remaps the shapes the guitarist actually plays. The core behavior follows the OnSong/OpenSong lineage rather than strict ChordPro — in ChordPro's spec the `{key}` directive names the shape key and capo only informs the listener's sounding key, whereas SongBook Pro's docs clearly treat the Key dropdown as the sounding/target key and the Capo value as a display-time reduction of chord shapes. A few details remain unverified by public sources and warrant caution in any parser implementation: **whether the `KeyShift` range is clamped** (likely ±11 or ±6 but not confirmed), **exactly how the major/minor distinction is stored** (very likely a separate flag adjacent to `key`, given minor support landed only in 2019–2020), and **whether the enharmonic spelling** (B♭ vs. A♯) is inferred from context or stored as an additional flag. None of these affect the central model above, and none block a working read/write implementation of the format — they only surface at edge cases the user explicitly excluded from scope.
